declaration_or_ruleset()

This is a nasty hack, and the only place in the parser that requires backtracking. The reason is that we can’t figure out if certain strings are declarations or rulesets with fixed finite lookahead. For example, “foo:bar baz baz baz...“ could be either a property or a selector.

To handle this, we simply check if it works as a property (which is the most common case) and, if it doesn’t, try it as a ruleset.

We could eke some more efficiency out of this by handling some easy cases (first token isn’t an identifier, no colon after the identifier, whitespace after the colon), but I’m not sure the gains would be worth the added complexity.

     # File lib/sass/scss/parser.rb, line 951
951:       def tok(rx, last_group_lookahead = false)
952:         res = @scanner.scan(rx)
953:         if res
954:           # This fixes https://github.com/nex3/sass/issues/104, which affects
955:           # Ruby 1.8.7 and REE. This fix is to replace the ?= zero-width
956:           # positive lookahead operator in the Regexp (which matches without
957:           # consuming the matched group), with a match that does consume the
958:           # group, but then rewinds the scanner and removes the group from the
959:           # end of the matched string. This fix makes the assumption that the
960:           # matched group will always occur at the end of the match.
961:           if last_group_lookahead && @scanner[-1]
962:             @scanner.pos -= @scanner[-1].length
963:             res.slice!(-@scanner[-1].length..-1)
964:           end
965:           @line += res.count(NEWLINE)
966:           @expected = nil
967:           if !@strs.empty? && rx != COMMENT && rx != SINGLE_LINE_COMMENT
968:             @strs.each {|s| s << res}
969:           end
970:           res
971:         end
972:       end
